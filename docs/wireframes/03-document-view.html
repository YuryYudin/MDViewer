RFC-031: Per-tenant Rate Limiting

Status: draft · Author: Priya Iyer · Updated 2026-04-22

Summary

Today the gateway enforces a single global request budget. As we onboard larger customers, a single tenant's traffic spike can degrade response times for every other tenant. This RFC proposes a per-tenant token-bucket limiter at the gateway.

Goals

  • Bound the worst-case impact one tenant can have on shared infrastructure.
  • Expose limit headroom to tenants via response headers.
  • Operate without adding a synchronous round-trip to a central counter.

Design

Each gateway pod owns a local token bucket per (tenant, route-class) pair, replenished from a global ledger every refresh_interval seconds. The global ledger lives in Redis and is updated atomically using a Lua script.

def consume(tenant: str, route_class: str, n: int = 1) -> bool:
    bucket = local_buckets[(tenant, route_class)]
    if bucket.tokens >= n:
        bucket.tokens -= n
        return True
    return False  # 429 returned to client

Why per-pod buckets, not per-edge

Per-edge buckets concentrate state and increase contention. Per-pod buckets trade some accuracy for resilience and horizontal scalability.

Operational concerns

ConcernMitigation
Unbalanced podsSticky tenant routing at L7
Cold-start lagPods preload bucket state at boot
Lua script driftVersioned and shipped via config service

  1. How do we communicate budget exhaustion to billing in real time?
  2. Should burst credit accrue overnight for low-traffic tenants?
