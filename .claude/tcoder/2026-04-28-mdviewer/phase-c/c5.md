# C5: E2E suite green (verification only)

**Avoid:**
- Do NOT add production code in this task. WHY: this is the outer green of double-loop TDD. If a spec needs a code change, the change belongs in the upstream task (A8/A10/A11/B4/C2/C3 etc.) and that task's status is wrong. Treat any failing spec here as a regression report against an earlier task.
- Do NOT loosen assertions in failing specs. WHY: the A1 RED specs were written against wireframe-derived selectors; weakening them to make them pass produces a green CI that does not actually verify the design's success criteria.
- Do NOT skip the coverage gate. WHY: the design's Test Coverage section sets the threshold to 90 for both Rust (`cargo llvm-cov`) and frontend (`npm run test:coverage`); skipping this would let the project ship with under-tested modules.
- Do NOT bypass `tauri build` and run the e2e against `npm run dev`. WHY: dev mode bypasses the bundling pipeline and would mask any regression introduced by C4. Run the e2e suite against the built MDViewer binary the way a user runs it.

## Steps

### Step 1: Build the app for e2e

```bash
npm run tauri -- build --debug
```

Confirm `src-tauri/target/debug/mdviewer` (`mdviewer.exe` on Windows) exists. The A1 helper points at this binary.

### Step 2: Run the full WebdriverIO + tauri-driver suite

```bash
npm run test:e2e
```

Each spec must pass. The expected PASS list:

1. `01-open-render.spec.ts`
2. `02-profile-setup.spec.ts`
3. `03-add-comment.spec.ts`
4. `04-reply-resolve.spec.ts`
5. `05-edit-reattach.spec.ts`
6. `06-auto-merge.spec.ts`
7. `07-conflict-diff.spec.ts`
8. `08-share-export.spec.ts`
9. `09-settings-take-effect.spec.ts`
10. `10-theme-switch.spec.ts`

If a spec fails, identify the upstream task whose `done_when` it covers and treat the failure as a regression bug against that task — do not patch from C5.

### Step 3: Verify coverage thresholds

```bash
cd src-tauri && cargo llvm-cov --workspace --html
cd .. && npm run test:coverage
```

Both must report >= 90% on touched modules / files. Coverage HTML reports land at `src-tauri/target/llvm-cov/html/` and `coverage/index.html` respectively for inspection.

If a module is below threshold:
- Add tests in that module's existing test file (e.g., `src-tauri/tests/anchor.rs`, `tests/views/Document.test.ts`) — not in a new C5-owned suite.
- Re-run the gate.

### Step 4: Sanity-check the bundled artifact (optional)

If on macOS:
```bash
open src-tauri/target/debug/bundle/macos/MDViewer.app
```

Drive the app manually for a smoke pass against wireframe 03 (open a markdown file, see it rendered with at least one comment). Repeat the same on Windows / Linux runners as the release workflow exercises.

### Step 5: Commit (no production changes)

This task introduces no new files. If everything passes there is nothing to commit — record the pass in the plan's status by closing the task. If a coverage gap forced a test-only addition under one of the existing test files, commit it under the upstream task's ID rather than as C5.

In short: C5 is a verification gate. The "done_when" is satisfied by green output, not by source changes.
