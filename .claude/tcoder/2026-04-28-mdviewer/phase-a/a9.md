# A9: Frontend shell, theme, IPC adapter, StartPage, ProfileSetup

**Avoid:**
- Do NOT call `@tauri-apps/api`'s `invoke` directly from each view. WHY: Vitest's jsdom environment cannot resolve the Tauri bridge, and every view test would have to mock the global. Centralize IPC in `src/ipc.ts` so view tests inject a fake adapter.
- Do NOT use a frontend framework (React, Solid, Vue). WHY: the design names "vanilla TS view layer". Pulling in a framework would balloon the WebView bundle and undermine the "small bundle" rationale that drove the Tauri-over-Electron decision.
- Do NOT write theme classes onto every element. WHY: the wireframes use CSS variables — `--bg`, `--fg`, `--accent`, etc. — so a single class on `<body>` re-themes every surface. The Settings to consumer map says theme switches must be honored everywhere; a class-on-body approach is the only way that scales.
- Do NOT block the first paint on settings load. WHY: settings load reaches across the IPC boundary; if we await it before the StartPage mounts, the user sees a blank window for ~100ms. Mount StartPage with the saved theme already applied (cached from `localStorage` as a hint) and reconcile when settings arrive.

### Step 2: Implement IPC adapter

Create `src/ipc.ts`. Type definitions are NOT hand-written — they come from `src/types-generated.ts`, which is produced by `cargo run --bin export_types` (A2b). Re-export the relevant types so views can import from a single module:

```ts
import { invoke } from '@tauri-apps/api/core';

// Re-export every IPC type from the generated file. Hand-writing them here
// would re-introduce the Rust↔TS drift that broke earlier plan iterations.
export type {
  BuildInfo,
  ProfileSettings, AppearanceSettings, EditorSettings, CommentsSettings, AdvancedSettings,
  Settings, Theme, ExternalChangeBehavior, AutoMergeMode,
  RenderOptions, RenderResult,
  Anchor, ResolveOutcome,
  Comment, Thread, NewComment, NewThread,
  OpenResult, OpenOutcome,
} from './types-generated';

import type {
  BuildInfo, Settings, Anchor, ResolveOutcome, Thread, OpenOutcome, RenderResult,
} from './types-generated';

// `Anchor` is the canonical name across the wire. Older planning notes used
// `AnchorPayload`; if you see references to it, alias here for clarity:
export type AnchorPayload = Anchor;

export interface Ipc {
  appInfo(): Promise<BuildInfo>;
  openDocument(path: string): Promise<OpenOutcome>;
  closeTab(id: string): Promise<void>;
  activateTab(id: string): Promise<void>;
  listOpenDocuments(): Promise<string[]>;
  listRecents(): Promise<string[]>;
  getSettings(): Promise<Settings>;
  setSettings(s: Settings): Promise<void>;
  listThreads(tabId: string): Promise<Thread[]>;
  createThread(tabId: string, anchor: Anchor, body: string): Promise<Thread>;
  postReply(tabId: string, threadId: string, body: string): Promise<void>;
  resolveThread(tabId: string, threadId: string): Promise<void>;
  renderMarkdown(source: string): Promise<RenderResult>;
  resolveAnchor(tabId: string, anchor: Anchor): Promise<ResolveOutcome>;
}

export const tauriIpc: Ipc = {
  appInfo: () => invoke('app_info'),
  openDocument: (path) => invoke('open_document', { path }),
  closeTab: (id) => invoke('close_tab', { id }),
  activateTab: (id) => invoke('activate_tab', { id }),
  listOpenDocuments: () => invoke('list_open_documents'),
  listRecents: () => invoke('list_recents'),
  getSettings: () => invoke('get_settings'),
  setSettings: (s) => invoke('set_settings', { settings: s }),
  listThreads: (tabId) => invoke('list_threads', { tabId }),
  createThread: (tabId, anchor, body) => invoke('create_thread', { tabId, anchor, body }),
  // post_reply / resolve_thread take tab_id explicitly (the Rust handler uses

  postReply: (tabId, threadId, body) => invoke('post_reply', { tabId, threadId, body }),

  resolveAnchor: (tabId, anchor) => invoke('resolve_anchor', { tabId, anchor }),
```

### Step 4: Workspace, TabBar, StartPage, ProfileSetup views

Create views that build DOM via `createElement` + `textContent` (avoiding raw HTML injection where any user data is involved). For static skeletons, `<template>` parsing is acceptable since the source string is a code-controlled literal.

Create `src/views/Workspace.ts`:

```ts
import type { Ipc } from '../ipc';
import { mountStartPage } from './StartPage';
import { mountTabBar } from './TabBar';

export interface WorkspaceState {
  tabs: { id: string; path: string }[];
  activeId: string | null;
}

export async function mountWorkspace(root: HTMLElement, ipc: Ipc): Promise<{ refresh: () => Promise<void> }> {
  // Static skeleton, no user data — built via createElement to keep view tests
  // simple and to keep us aligned with the no-innerHTML house style.
  root.replaceChildren();
  const shell = document.createElement('div');
  shell.setAttribute('data-view', 'workspace');
  shell.className = 'workspace';
  for (const region of ['titlebar', 'tabbar', 'body', 'status']) {
    const el = document.createElement('div');
    el.setAttribute('data-region', region);
    shell.appendChild(el);
  }
  root.appendChild(shell);

  const state: WorkspaceState = { tabs: [], activeId: null };
  const tabbar = shell.querySelector<HTMLElement>('[data-region="tabbar"]')!;
  const body = shell.querySelector<HTMLElement>('[data-region="body"]')!;

  async function refresh() {
    const ids = await ipc.listOpenDocuments();
    state.tabs = ids.map((id) => ({ id, path: id }));
    mountTabBar(tabbar, ipc, state);
    if (state.tabs.length === 0) {
      await mountStartPage(body, ipc);
    } else {
      // Document/Conflict/Settings views are mounted by A10/A11/C2.
      body.replaceChildren();
      const placeholder = document.createElement('div');
      placeholder.setAttribute('data-view', 'document');
      body.appendChild(placeholder);
    }
  }
  await refresh();
  return { refresh };
}
```

Create `src/views/StartPage.ts`:

```ts
import type { Ipc } from '../ipc';

export async function mountStartPage(root: HTMLElement, ipc: Ipc): Promise<void> {
  root.replaceChildren();
  const view = document.createElement('section');
  view.setAttribute('data-view', 'start');

  const open = document.createElement('button');
  open.setAttribute('data-action', 'open-file');
  open.textContent = 'Open…';
  view.appendChild(open);

  // Production: open the native file dialog via @tauri-apps/plugin-dialog.
  // E2E (WebdriverIO): also expose a hidden <input type="file"> that the
  // test driver can call setValue / uploadFile against, since tauri-driver
  // cannot drive the OS dialog. The visible Open button uses the dialog in
  // production; the hidden input is gated on the e2e env var so production
  // users never see two paths racing.
  open.addEventListener('click', async () => {
    if (import.meta.env?.MDVIEWER_E2E === '1') {
      // Tests trigger the input directly; clicking the button just opens it
      // for keyboard users in e2e mode.
      fileInput.click();
      return;
    }
    const { open: openDialog } = await import('@tauri-apps/plugin-dialog');
    const picked = await openDialog({
      filters: [{ name: 'Markdown', extensions: ['md', 'markdown'] }],
      multiple: false,
    });
    if (typeof picked === 'string') await ipc.openDocument(picked);
  });

  const fileInput = document.createElement('input');
  fileInput.type = 'file';
  fileInput.accept = '.md,.markdown,text/markdown';
  fileInput.setAttribute('data-test', 'file-input');
  fileInput.style.display = 'none';
  fileInput.addEventListener('change', async () => {
    const f = fileInput.files?.[0];
    if (!f) return;
    // Under WebdriverIO + tauri-driver, `browser.uploadFile(...)` returns an
    // absolute path that the runner's setValue places into the input's value
    // field; reading it back gives the absolute path. In a normal browser
    // File picker the value would be a basename only, but in production we
    // never reach this handler because the click handler above takes the
    // dialog path.
    const path = (fileInput.value || (f as unknown as { path?: string }).path || f.name);
    await ipc.openDocument(path);
  });
  view.appendChild(fileInput);

  const settings = document.createElement('button');
  settings.setAttribute('data-action', 'open-settings');
  settings.textContent = 'Settings';
  view.appendChild(settings);

  const recents = await ipc.listRecents();
  const list = document.createElement('ul');
  list.setAttribute('data-test', 'recents');
  for (const path of recents) {
    const li = document.createElement('li');
    li.setAttribute('data-test', 'recent-item');
    li.textContent = path; // textContent prevents path-based markup injection
    li.addEventListener('click', () => { ipc.openDocument(path); });
    list.appendChild(li);
  }
  view.appendChild(list);
  root.appendChild(view);
}
```

### Step 5b: Wire the keymap from settings.shortcuts

The Settings store carries a `shortcuts` table (keyed by action). The frontend reads it and attaches a single `keydown` listener that dispatches actions. v1 surfaces a fixed set of actions only; remap UI is a non-goal per the design.

```ts
// src/keymap.ts
import type { Settings } from './ipc';

// Action union must match A3's `default_shortcuts()` keys exactly.
export type Action =
  | 'open_file' | 'save_file' | 'toggle_edit' | 'close_tab'
  | 'comment_on_selection' | 'toggle_sidebar' | 'resolve_thread'
  | 'toggle_dark' | 'open_settings';

export type ActionHandler = (a: Action) => void;

export function installKeymap(settings: Settings, handler: ActionHandler): () => void {
  const lookup = new Map<string, Action>();
  for (const [action, combo] of Object.entries(settings.shortcuts)) {
    lookup.set(canonical(combo), action as Action);
  }
  const onKeyDown = (ev: KeyboardEvent) => {
    const key = canonicalFromEvent(ev);
    const action = lookup.get(key);
    if (action) { ev.preventDefault(); handler(action); }
  };
  window.addEventListener('keydown', onKeyDown);
  return () => window.removeEventListener('keydown', onKeyDown);
}

// Both sides — the user-visible combo string from settings AND the
// KeyboardEvent — are normalised to the SAME canonical form so lookup
// hits. Form: lowercase tokens, sorted alphabetically, joined by '+'.
// The platform-meta token "mod" maps to Cmd on macOS, Ctrl elsewhere; the
// settings string uses "Mod" or "CmdOrCtrl" interchangeably and both
// resolve to the same canonical "mod".

function canonical(combo: string): string {
  return tokenize(combo).map(t => mapModToken(t)).sort().join('+');
}

function canonicalFromEvent(ev: KeyboardEvent): string {
  const parts: string[] = [];
  if (ev.metaKey || ev.ctrlKey) parts.push('mod');
  if (ev.shiftKey) parts.push('shift');
  if (ev.altKey) parts.push('alt');
  parts.push(ev.key.toLowerCase());
  return parts.sort().join('+');
}

function tokenize(combo: string): string[] {
  return combo.toLowerCase().split('+').map(s => s.trim()).filter(Boolean);
}

function mapModToken(t: string): string {
  if (t === 'cmdorctrl' || t === 'cmd' || t === 'ctrl' || t === 'meta') return 'mod';
  return t;
}
```

Add a Vitest case to `tests/keymap.test.ts` that asserts `canonical('Mod+Shift+M') === canonical from KeyboardEvent({metaKey:true, shiftKey:true, key:'M'})`. That keeps the contract verified at the unit level.

In `src/main.ts` after `await mountWorkspace(root, tauriIpc);`, install the keymap. Both `currentTheme` and the dispatcher live INSIDE `main()` so they close over the local `settings`.

```ts
import { installKeymap, type Action } from './keymap';

// Inside main(), after `const settings = await tauriIpc.getSettings();`:
let currentTheme: 'light' | 'dark' | 'follow_system' = settings.appearance.theme;

const applyTheme = (theme: 'light' | 'dark' | 'follow_system') => {
  currentTheme = theme;
  const dark = theme === 'dark' ||
    (theme === 'follow_system' && window.matchMedia('(prefers-color-scheme: dark)').matches);
  document.body.classList.toggle('theme-dark', dark);
  localStorage.setItem('mdviewer.theme', dark ? 'dark' : 'light');
};
applyTheme(currentTheme);

const dispatchAction = (action: Action) => {
  switch (action) {
    case 'open_file': document.querySelector<HTMLInputElement>('[data-test="file-input"]')?.click(); break;
    case 'save_file': document.dispatchEvent(new CustomEvent('mdviewer:save-active')); break;
    case 'toggle_edit': document.dispatchEvent(new CustomEvent('mdviewer:toggle-edit')); break;
    case 'comment_on_selection': document.dispatchEvent(new CustomEvent('mdviewer:comment-on-selection')); break;
    case 'toggle_sidebar': document.dispatchEvent(new CustomEvent('mdviewer:toggle-sidebar')); break;
    case 'resolve_thread': document.dispatchEvent(new CustomEvent('mdviewer:resolve-focused-thread')); break;
    case 'toggle_dark': applyTheme(currentTheme === 'dark' ? 'light' : 'dark'); break;
  }
};
const _uninstall = installKeymap(settings, dispatchAction);

// End of main() body. Remove the standalone `applyTheme` block earlier in
// the file — it lived at module scope and referenced `settings`, which is
// only valid inside this function.
```

Each view (`Document.ts`, `Edit.ts`, `CommentsSidebar.ts`, `ThreadDetail.ts`, `Workspace.ts`) listens for the corresponding `mdviewer:*` custom event on `document` and reacts when it owns the focus. This keeps the keymap a thin dispatcher and avoids cyclic imports between views and the bootstrapper.

A unit test in `tests/keymap.test.ts` should assert: (a) `Cmd+O` (or its serialized form) triggers `open_file`; (b) unknown combos do not dispatch; (c) the returned cleanup function removes the listener.

This satisfies the "Shortcut bindings → frontend keymap in `src/main.ts`" mapping in the design's Settings → consumer table. The remap UI itself remains a non-goal in v1.
